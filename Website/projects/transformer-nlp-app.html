Transformer-based NLP applications

A two-part assignment covering three distinct transformer applications: a retrieval-based question-answering system, automated translation of category labels, and a fine-tuned text classifier — built across one CPU and one GPU notebook.

ML & NLP Python Teamwork

The three tasks

Task 1 — CPU

Retrieval-based question answering

A chatbot that answers questions about a single source document — in our case, the original BERT paper — by retrieving the most relevant passage before extracting an answer from it, rather than trying to answer from the model's general knowledge alone.

  • Split the document into overlapping 3-sentence chunks using a sliding window, so context isn't lost at chunk boundaries.
  • Encoded each chunk into a vector with a sentence-transformer model, and encoded incoming questions the same way.
  • Retrieved the top-K most similar chunks to a given question, discarding any below a similarity threshold.
  • If no chunk cleared the threshold, the system returned "I don't think I can answer this question" rather than guessing.
  • Concatenated the surviving chunks into a single context string, then used a question-answering pipeline to extract the actual answer span.

We experimented with different values for the similarity threshold and top-K, since both directly affect whether the system answers confidently, answers wrongly, or correctly declines.

Task 2 — CPU

Category translation

The classification task (below) uses a dataset of 14,000 Simplified Chinese news articles spread across 14 categories. Before classifying anything, we translated the category labels themselves into English using a pretrained translation model, then sanity-checked the results against an online translation service.

Task 3 — GPU

News article classification

Fine-tuned a pretrained Chinese-language transformer to classify a news article into one of the 14 categories — going from a general-purpose language model to a task-specific classifier.

  • Split the dataset 80/10/10 into train/eval/test, stratified by category to keep each split balanced.
  • Tokenised all articles and built dataset objects for train, eval, and test.
  • Fine-tuned for 3 epochs at a learning rate of 2e-5, evaluating accuracy, precision, and recall after each epoch.
  • Evaluated final performance on the held-out test set and produced a confusion matrix to see which categories were most often confused with each other.
Geotrend/distilbert-base-zh-cased

Results

Both the QA system and the classifier performed sensibly — the QA system gave reasonable answers for in-scope questions and correctly declined questions it had no basis to answer, and the fine-tuned classifier learned to distinguish between the 14 news categories.

Note: we didn't formally log the final accuracy/precision/recall numbers or export the confusion matrix from this run — if I revisit the notebook, I'd add the exact metrics here.

What I'd do differently

I'd log and save evaluation metrics and the confusion matrix at the time of training, rather than relying on memory afterward — small thing, but it's the difference between being able to say "it worked" and being able to say exactly how well.
